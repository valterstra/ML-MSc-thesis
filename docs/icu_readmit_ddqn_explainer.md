## 1. What DDQN is doing here

DDQN is the **control policy** that sits on top of CARE-Sim.

- Step 11a trains the simulator: `state/action/history -> next state`
- Step 13 learns the policy: `observation -> action value`

So DDQN does not learn patient dynamics. The transformer simulator already does
that. DDQN learns which action should be chosen now so that future reward is as
good as possible.

## 2. What the Q-network takes in and gives out

The Q-network input is one observation vector built from the recent history of
state/action pairs.

In the selected Step 13 setup:

- observation window length = `5`
- state dimension = `9`
- action dimension = `5`

So the input size is:

```text
5 * (9 + 5) = 70
```

The Q-network output is:

- `32` numbers
- one Q-value for each possible action combination

So the network answers:

```text
Q(observation, action_id)
```

for all `action_id in {0, ..., 31}`.

## 3. What the Q-value means

A Q-value is not a probability.

It is the network's estimate of:

```text
how much total future reward I expect if I take this action now
```

So the best action is simply:

```text
argmax_a Q(observation, a)
```

## 4. What one training step looks like

One DDQN training sample is:

```text
(o, a, r, o_next, done)
```

where:

- `o` = current observation
- `a` = action chosen at that observation
- `r` = reward returned by the simulator
- `o_next` = next observation after that action
- `done` = whether the rollout ended

### Where the sample comes from

1. Start from a real seed history.
2. Build the current observation `o`.
3. Use `Q_online` to choose an action, sometimes random, sometimes greedy.
4. Pass that action into CARE-Sim.
5. The simulator returns `r`, `o_next`, and `done`.
6. Store the transition in replay memory.

So the simulator creates the experience, and DDQN learns from that experience.

## 5. How the target is built

DDQN uses two networks:

- `Q_online`: updated every training step
- `Q_target`: a slower copy used to build stable targets

For one sample `(o, a, r, o_next, done)`:

1. Read the current prediction:

```text
q_current = Q_online(o, a)
```

2. Build the target:

DDQN does **not** use the target network by itself. It uses the online and target
networks together:

- `Q_online(o, a)` gives the current value estimate for the action that was taken
- `Q_online(o_next, ·)` is used to choose the best next action
- `Q_target(o_next, ·)` is used to evaluate that next action

So:

- the online network chooses the next action
- the target network evaluates it

The target is:

```text
target = r + gamma * (1 - done) * Q_target(o_next, argmax_a Q_online(o_next, a))
```

If `done = 1`:

```text
target = r
```

Otherwise:

```text
best_next_action = argmax_a Q_online(o_next, a)
target = r + gamma * Q_target(o_next, best_next_action)
```

3. Update `Q_online` so that `q_current` moves closer to `target`.

That comparison is what drives learning. In code, the update minimizes the
difference between `q_current` and `target` with a regression loss.

That is the whole DDQN update.

## 6. Why there are two networks

This is the main DDQN stabilizer.

- `Q_online` changes fast
- `Q_target` changes slowly

Why:

- if the same network both chooses the next action and scores it, training can
  become unstable
- the delayed target makes the learning signal less noisy

So:

- `Q_online` chooses the next action for the target
- `Q_target` evaluates that action

`Q_target` is not trained directly. It is updated by copying weights from
`Q_online` every so often.

## 7. Where the simulator fits

The simulator is the environment.

At each step:

1. DDQN picks an action.
2. CARE-Sim predicts the next state.
3. CARE-Sim returns a reward.
4. CARE-Sim returns whether the episode ended.

So the counterfactual question lives in the simulator:

- "What happens if I take action A instead of action B from the same history?"

DDQN itself does not explicitly enumerate all counterfactual futures. It learns
from many simulated transitions generated by the simulator.

## 8. Where counterfactuals matter

Counterfactuals are used in two places:

- planner evaluation: try multiple actions from the same patient context and
  compare simulated futures
- policy learning: DDQN learns from many simulated transitions, so it can
  assign higher value to actions that lead to better futures

So the simulator is the counterfactual engine. DDQN is the policy learner built
on top of it.

## 9. What reward DDQN is optimizing

In the current selected Step 13 setup, the environment reward is:

- dense reward from handcrafted severity improvement
- terminal reward from the readmission model
- uncertainty penalty when the simulator is too uncertain

So DDQN is not trying to maximize next-state quality only.
It is learning a short-horizon policy that balances:

- better physiology
- better final readmission outlook
- staying inside the simulator's reliable regime

## 10. What training produces at the end

After training, you get one learned policy network.

At inference time:

1. Build the current 70-dimensional observation.
2. Feed it into `Q_online`.
3. Get 32 Q-values.
4. Pick the action with the highest value.
5. Decode that action ID into the 5 binary ICU actions.

So the final artifact is a **policy**:

```text
recent history -> recommended next action
```

## 11. One important limitation

DDQN is only as good as:

- the simulator dynamics
- the reward design
- the finite rollout horizon

In this project, that is why the Step 13 horizon is short.
It is a finite-horizon control problem, not a full natural-length ICU episode
optimizer.

## 12. The short version

If you want the whole algorithm in one paragraph:

DDQN repeatedly starts from a real seed history, chooses an action using the
current Q-network, sends that action to CARE-Sim, stores the resulting
transition, and trains the online Q-network so that its predicted action value
matches the simulator-based target `r + gamma * future_value`. The target
network is just a delayed copy that makes this update stable.
